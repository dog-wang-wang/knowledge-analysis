# 1. 目标
分析每一个文件的作用，并在目标的输出目录中输出每个文件的分析html与入口的总结html

# 2. 工作流
## 2.1. 确定输出的目标目录
查看[config.properties](../scripts/config.properties)获取其中的struct.root属性的参数值 
在项目的根目录中有和上述参数值相同的目录，如果没有，请创建一个。并将其作为目标目录。 
## 2.2. 创建analysis目录
在提取出的目标目录下创建名为analysis目录
## 2.3. 开始分析
### 分析流程
1. 遍历项目的所有文件
2. 分析每一个文件的作用，并且在analysis目录下创建每一个文件的分析结果的html。html文件的相关名称要参考原本的路径结构，包括分包也要参考原来的路径结构
3. 最后创建一个入口html，不但可以索引到刚才创建的html，内部的内容还需要能够串联起文件的关系。相当于一个缩略版的总结文档。
4. 检查是否有遗漏的未分析或未输出分析文件的文件。如果有，必须补齐。直至验收时无遗漏文件
5. 校验是否白名单内的文件是否都生成了相应的分析html，没有就继续分析这些白名单内未被分析的文件
6. 检验入口html是否能够索引到生成的所有html，如果没有，要补全。

# 4. 要求
## 4.1. 分析范围：
除了上述黑名单目录下的文件外。每个文件都需要分析并形成分析html，任何类型的文件都需要分析
## 4.2. 分析方法：
1. 分析结果一定要详细，包括但不限于整个文件的作用
2. 如果被分析的文件是代码。那么这个文件内的每一个方法都要分析到位。包括但不限于方法的作用。方法的参数，参数的含义，方法的调用链。此外，如果内部含有其他的类，一定要详细分析。这些内容全部都要展现在该文件分析结果的html中
3. 分析结果的html中右边一定要专门有一块区域用来讲解该文件涉及到的专业术语并且用通俗的大白话总结一下这个分析文档
## 4.3. 入口HTML文件要求：
1. 该文件应该有从项目入口函数开始的完整调用链关系，并以清晰直观的方式呈现出来。对于每一步要有详细的中文解释。
2. 该文件需要能够索引到所有的分析结果文件。
3. 内部的内容还需要能够串联起文件的关系。相当于一个缩略版的总结文档。还需要加入知识图谱，展示每个文件的关系。
4. 该文件内部需要展示从项目入口函数开始的整个调用链。可以用时序图也可以流程图。要求必须人类可以清晰直观的了解整个流程。
## 4.4. 红线
1. 白名单中的任何文件都有相同的重要性。必须都必须全量分析，不能厚此薄彼。不能遗漏，如有遗漏必须补充。不能编造假的内容
2. 任何格式的文件都必须考虑在内，哪怕无后缀文件也需要分析。不能出现白名单内的文件未被考虑的情况。
3. 生成的html必须保证能够在浏览器中正常显示。编码无乱码
