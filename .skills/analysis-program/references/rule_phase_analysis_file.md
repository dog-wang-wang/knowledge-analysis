# 目标
分析每一个文件的作用，并在目标的输出目录中输出每个文件的分析html与入口的总结html

# 工作流
## 确定输出的目标目录
查看[config.properties](../scripts/config.properties)获取其中的struct.root属性的参数值 
在项目的根目录中有和上述参数值相同的目录，如果没有，请创建一个。并将其作为目标目录。 
## 创建analysis目录
在提取出的目标目录下创建名为analysis目录
## 开始分析
### 流程
1. 遍历项目的所有文件
2. 分析每一个文件的作用，并且在analysis目录下创建每一个文件的分析结果的html。html文件的相关名称要参考原本的路径结构，包括分包也要参考原来的路径结构
3. 最后创建一个入口html，不但可以索引到刚才创建的html，内部的内容还需要能够串联起文件的关系。相当于一个缩略版的总结文档。
4. 检查是否有遗漏的未分析或未输出分析文件的文件。如果有，必须补齐。直至验收时无遗漏文件
5. 校验是否白名单内的文件是否都生成了相应的分析html，没有就继续分析这些白名单内未被分析的文件
6. 检验入口html是否能够索引到生成的所有html

# 要求
1. 不需要分析目标目录下的文件
2. 不需要分析.skills目录下的文件
3. 除了上述黑名单外，每个文件都需要分析并形成分析html，任何类型都需要分析
4. 针对单个文件的分析结果一定要详细，包括但不限于整个文件的作用，如果是编程语言，每个函数的的作用，参数及其含义，可能调用的地点都要详细描述出来
5. 针对单个文件的分析，内部在页面的右边一定要专门有一块区域用来讲解该文件涉及到的专业术语并且用通俗的大白话总结一下这个分析文档
6. 入口html，不但可以索引到刚才创建的每一个html，内部的内容还需要能够串联起文件的关系。相当于一个缩略版的总结文档。还需要加入知识图谱，展示每个文件的关系
7. 忽略掉文件重要与否的观念，所有的文件同等重要，所有白名单内的文件都必须分析，不能遗漏。如果遗漏必须补充
8. 任何格式的文件都必须考虑在内，哪怕无后缀也要分析。禁止出现像是txt，list，jpg之类的文件不被考虑的情况
9. 生成的html不可以有任何乱码。
